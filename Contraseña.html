Extracción Segura de Datos: Lambda

1. El Diagrama de Flujo del Proceso 🚀

Tu objetivo es mover datos de personas desde una base de datos a un archivo CSV en un *bucket* de S3, asegurando todo el trayecto.

Base de Datos (Origen)
➡️
AWS Secrets Manager
(Credenciales Seguras)
➡️
AWS Lambda (Ejecución y DataFrame)
➡️
Amazon S3 Bucket
(CSV Cifrado con KMS)

2. ¿Cómo NO Guardar Datos Sensibles?

¡NUNCA PONGAS LAS CREDENCIALES DIRECTAMENTE EN EL CÓDIGO!

El error más común es almacenar secretos (contraseñas, llaves API) en:

  • El archivo de código (`.py`) de Lambda.
  • Variables de entorno de Lambda, a pesar de estar cifradas por AWS.

Esto viola el principio de **separación de configuración y código** y expone tus secretos si alguien accede al código o a la configuración de la función.

3. La Manera Correcta de Guardar Datos Sensibles (Credenciales)

Para la conexión de tu Lambda a la Base de Datos, usarás **AWS Secrets Manager**.

Paso 1: Almacenar la Contraseña de la DB

Guarda el usuario, contraseña y host de tu base de datos en **AWS Secrets Manager**. Este servicio está diseñado para almacenar, rotar y gestionar secretos de forma segura.

Paso 2: Usar Variables de Entorno de Forma Segura

En las **Variables de Entorno** de tu función Lambda, solo guarda el **ARN** (Amazon Resource Name) del secreto, ¡no el secreto en sí!

VARIABLE_ENTORNO: SECRET_ARN
VALOR: arn:aws:secretsmanager:us-east-1:123456789012:secret:MiDBSecret-XXXXX

Paso 3: Mínimo Privilegio (IAM Role)

Tu función Lambda necesita un **Rol de IAM** que solo le dé permisos para **leer** ese secreto específico de Secrets Manager (secretsmanager:GetSecretValue).

4. Transformando y Asegurando el DataFrame (Lambda Code)

A. Anonimización y Seudonimización (Transformación de Datos Sensibles)

Si los datos de la "tabla de personas" contienen información muy sensible (ej: emails, DNI, números de teléfono), debes **transformarlos antes de guardarlos en S3**.

# Cargar datos de la DB a un DataFrame (usando Pandas)
df = pd.DataFrame(datos_extraidos)

# **¡Zona de Anonimización!**
# Reemplazar el número de teléfono por una versión con hash (seudonimización)
df['telefono_hash'] = df['telefono'].apply(lambda x: hashlib.sha256(x.encode()).hexdigest())
df = df.drop(columns=['telefono']) # ⚠️ **Eliminar la columna original sensible**

B. Creación y Carga del CSV

El DataFrame se convierte en un objeto en memoria (CSV) y luego se sube a S3.

# Convertir DataFrame a CSV en memoria (IO Buffer)
csv_buffer = io.StringIO()
df.to_csv(csv_buffer, index=False)

# Cargar a S3
s3_client.put_object(
Bucket=os.environ.get('BUCKET_NAME'),
Key='datos/archivo_seguro.csv',
Body=csv_buffer.getvalue()
# **El cifrado se aplica automáticamente por la configuración del Bucket**
)

5. Protección Final en S3 (Datos en Reposo)

A. Cifrado Predeterminado del Bucket (SSE-KMS)

Esta es la defensa clave. Debes configurar el **Bucket de S3** para que **automáticamente cifre** cada archivo que se suba, utilizando **AWS KMS (Key Management Service)**.

**SSE-KMS** es el estándar de oro. AWS KMS gestiona las claves de cifrado, y tu archivo CSV sensible estará cifrado en todo momento. Solo las identidades (como tu Lambda) con permiso para usar la clave KMS podrán descifrarlo.

B. Bloqueo de Acceso Público

Asegúrate de que la configuración de **Bloqueo de Acceso Público** esté activada en el 100% de tus buckets para prevenir exposiciones accidentales.

C. Permisos de S3 (IAM Role)

Recuerda darle a tu Lambda permisos s3:PutObject solo en **ese bucket específico**, y permisos para usar la **clave KMS** que el bucket utiliza para cifrar.
